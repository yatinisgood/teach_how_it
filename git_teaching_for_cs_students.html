1. 什麼是靜態網站？

靜態網站是指主要由 HTML、CSS、JavaScript、圖片與其他前端檔案組成的網站。瀏覽器直接下載這些檔案並顯示，不需要後端伺服器即時計算資料。

類型 能不能用 GitHub Pages？ 例子
純 HTML / CSS / JS 可以 個人首頁、課堂作品、作品集、活動頁
Markdown / Jekyll 網站 可以 技術筆記、文件網站、Blog
React / Vue 打包後的靜態檔 可以，但通常要先 build 前端專案的 dist 或 build 輸出
Node.js / Python / PHP 後端服務 不適合 需要資料庫登入、API server、即時後端邏輯
新生先記這句

GitHub Pages 適合「前端靜態網站」，不適合拿來跑真正的後端程式。

2. GitHub Pages 是什麼？

GitHub Pages 是 GitHub 提供的靜態網站發布功能。你把網站檔案放在 repo 裡，再指定 GitHub Pages 要從哪個 branch、哪個資料夾讀取檔案，GitHub 就會幫你把它發布成一個網址。

本機電腦 ↓ git add / commit / push GitHub Repository ↓ GitHub Pages 讀取指定 branch + folder 公開網站網址

常見網址格式：

https://你的帳號.github.io/你的repo名稱/

例如你的 GitHub 帳號是 student123，repo 名稱是 my-first-site，那網站可能會是：

https://student123.github.io/my-first-site/

2.1 特殊情況：帳號首頁網站

如果 repo 名稱剛好是：

你的帳號.github.io

那它通常會變成帳號首頁，例如：

https://student123.github.io/

這比較適合拿來做個人作品集首頁。一般課堂練習則建議用普通 repo，例如 my-first-site。

3. Git、GitHub、Gitea 是什麼關係？

在教學生使用 GitHub Pages 之前，最好先分清楚：Git 不是 GitHub，GitHub 也不是 Git 本身。

最簡單的比喻

Git 是版本控制工具，負責記錄檔案修改歷史；GitHub / GitLab / Gitea 是放 Git repo 的平台，負責提供遠端倉庫、網頁介面、權限管理、協作功能。

名稱 它是什麼？ 通常負責什麼？ 可以自己架設嗎？
Git 分散式版本控制系統 本機版本紀錄、commit、branch、merge、push、pull Git 本身是工具，可以安裝在自己的電腦或伺服器
GitHub 雲端 Git 代管平台 遠端 repo、網頁瀏覽程式碼、issue、pull request、GitHub Pages、Actions 一般使用 github.com；企業版另有 GitHub Enterprise
Gitea 可自架的 Git 服務 遠端 repo、帳號權限、網頁介面、code review、issue、CI/CD 等 可以，例如安裝在 NAS、Docker、家用伺服器或公司內網

3.1 Git 是本體：沒有 GitHub 也可以用 Git

只要電腦有安裝 Git，就可以在本機資料夾做版本控制：

git init
git add .
git commit -m "Add first version"

這些動作即使沒有 GitHub、沒有網路，也可以完成。因為 commit 是先記錄在本機 repo 裡。

你的電腦 └── 專案資料夾 └── .git/ └── commit 歷史紀錄 這一層只需要 Git，不需要 GitHub。

3.2 GitHub / Gitea 是遠端平台：讓 repo 可以同步與分享

當你想把本機 repo 備份到遠端、換電腦同步、或讓其他人協作時，就會設定 remote。

git remote add origin https://github.com/你的帳號/my-project.git
git push -u origin main

如果遠端平台是你 NAS 上的 Gitea，概念也一樣，只是網址不同，例如：

git remote add origin http://你的NAS位址:3000/帳號/my-project.git
git push -u origin main

或如果你的 Gitea 有設定 HTTPS / 網域，可能會像：

git remote add origin https://git.example.com/帳號/my-project.git
git push -u origin main
重點

git add、git commit 是 Git 的本機操作；git push、git pull 則會碰到遠端平台，例如 GitHub 或 Gitea。

3.3 同一個本機 repo 可以推到不同遠端

Git 的 remote 只是「遠端倉庫位置」。所以同一個專案，理論上可以推到 GitHub，也可以推到你的 Gitea。

查看目前遠端：

git remote -v

可能看到 GitHub：

origin  https://github.com/你的帳號/my-project.git (fetch)
origin  https://github.com/你的帳號/my-project.git (push)

也可能看到 Gitea：

origin  http://你的NAS位址:3000/帳號/my-project.git (fetch)
origin  http://你的NAS位址:3000/帳號/my-project.git (push)

如果想改 remote URL：

git remote set-url origin 新的repo網址

如果想同時保留 GitHub 與 Gitea，也可以另外新增一個 remote 名稱：

git remote add gitea http://你的NAS位址:3000/帳號/my-project.git

# 推到 GitHub
git push origin main

# 推到 Gitea
git push gitea main
給新手的建議

初學階段先用一個 remote 就好，例如 origin 指向 GitHub。等觀念熟了，再示範同一個 repo 同步到 GitHub 與 Gitea。

3.4 GitHub Pages 是 GitHub 額外提供的網站發布功能

這一篇教的是 GitHub Pages，所以必須使用 GitHub 的 Pages 功能。你的 NAS 上的 Gitea 可以代管 Git repo，但它本身不等於 GitHub Pages。

需求 GitHub NAS 上的 Gitea
保存 Git repo 可以 可以
用瀏覽器看程式碼 可以 可以
多人協作、權限管理 可以 可以
GitHub Pages 靜態網站發布 可以，這是 GitHub 功能 不是 GitHub Pages；若要發布網站，通常要另外搭 Nginx、Apache、Caddy、Cloudflare Pages、Netlify、Vercel 或 NAS Web Station 等服務

3.5 給學生的心智模型

Git：版本控制引擎 ├── 在本機 commit ├── 記錄誰改了什麼 └── 可以 push / pull 到遠端 GitHub：雲端 Git 平台 ├── 放遠端 repo ├── 管理 issue / PR / Actions └── 提供 GitHub Pages 發布靜態網站 Gitea：自架 Git 平台 ├── 放遠端 repo ├── 可裝在 NAS / Docker / 內網 └── 適合私人、家庭、公司內部代管程式碼

所以可以這樣總結：

  • Git 是技術標準與工具。
  • GitHub 是使用 Git 的雲端平台。
  • Gitea 是可以自己架設的 Git 平台。
  • GitHub Pages 是 GitHub 額外提供的靜態網站發布功能，不是 Git 本身的功能。

4. 為什麼要設定 Branch？

很多人第一次用 GitHub Pages 會卡在這裡：為什麼不是 repo 建好就自動變網站？為什麼還要去 Settings → Pages 選 branch？

原因很簡單

一個 repo 裡面可能有很多檔案，不一定全部都是網站要公開的內容。GitHub Pages 需要你明確告訴它：「請從哪個 branch、哪個資料夾拿檔案來當網站。」

3.1 Branch 是什麼？

branch 可以先簡單理解成「專案的一條版本線」。新手通常只會用到 main 這條主線。

main branch ├── index.html ├── style.css └── app.js GitHub Pages 設定： Source: Deploy from a branch Branch: main Folder: /root 結果：GitHub Pages 會從 main branch 的最上層資料夾讀取網站檔案。

3.2 為什麼常選 main branch？

因為 main 通常是 repo 的主要版本，也是新手最容易理解與維護的分支。對入門靜態網站來說，先把網站檔案直接放在 main branch 是最直覺的做法。

3.3 Folder 選 root 或 /docs 是什麼意思？

選項 意思 適合情境
/root repo 最外層資料夾 最簡單的 HTML / CSS / JS 小網站
/docs repo 裡的 docs 資料夾 repo 同時有原始碼與文件網站時
本篇建議

初學者先選 main + /root。等以後專案變複雜，再理解 /docs、gh-pages 或 GitHub Actions 部署。

5. 建議專案結構

4.1 最小網站結構

my-first-site/
├── index.html
├── style.css
└── script.js

其中 index.html 很重要。多數靜態網站服務會把 index.html 當作首頁。

4.2 有圖片時的結構

my-first-site/
├── index.html
├── style.css
├── script.js
└── images/
    └── profile.jpg

HTML 引用圖片時可以寫：

<img src="images/profile.jpg" alt="個人照片">

4.3 不建議的結構

my-first-site/
└── web/
    ├── index.html
    ├── style.css
    └── script.js

如果 GitHub Pages 設定是 main + /root，但你的 index.html 被放在 web/ 裡，首頁就可能找不到。新手先把 index.html 放在 repo 最外層最穩。

6. 方法一：用 GitHub 網頁建立靜態網站

這個方法不用命令列，適合第一次示範 GitHub Pages 的概念。

建立新的 Public Repo

到 GitHub 右上角按 + → New repository，建立一個 repo，例如：

my-first-site

Visibility 建議先選 Public，因為靜態網站本來就是要公開瀏覽。

新增 index.html

進入 repo 後按 Add file → Create new file，檔名輸入：

index.html

內容輸入：

<!DOCTYPE html>
<html lang="zh-Hant">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>我的第一個 GitHub Pages 網站</title>
</head>
<body>
  <h1>Hello GitHub Pages!</h1>
  <p>這是我用 Git 和 GitHub 建立的第一個靜態網站。</p>
</body>
</html>

Commit 新檔案

在頁面下方 commit message 寫：

Add website homepage

然後按 Commit changes。

啟用 GitHub Pages

到 repo 的 Settings → Pages。

在 Build and deployment 區塊選：

  • Source：Deploy from a branch
  • Branch：main
  • Folder：/root

按 Save。

等待發布完成

GitHub Pages 通常需要一點時間部署。完成後，Settings → Pages 會顯示網站網址。

這個設定代表什麼？

main + /root 的意思是：GitHub Pages 會從 main branch 的 repo 最上層讀取 index.html 作為網站首頁。

7. 方法二：用命令列建立靜態網站

這比較接近實際開發流程：在本機寫網站，commit 後 push 到 GitHub，再由 GitHub Pages 發布。

6.1 建立資料夾

mkdir my-first-site
cd my-first-site

6.2 建立 index.html

你可以用 VS Code 建立檔案，也可以先用以下方式快速建立：

echo ^<h1^>Hello GitHub Pages^</h1^> > index.html

若使用 Git Bash，也可以寫：

echo "<h1>Hello GitHub Pages</h1>" > index.html

6.3 初始化 Git 並 commit

建議第一次就把初始 branch 設成 main，避免本機和 GitHub 預設 branch 名稱不同。

git init -b main
git add .
git commit -m "Add initial static website"

如果你的 Git 版本不支援 git init -b main，可以改用：

git init
git branch -M main
git add .
git commit -m "Add initial static website"

6.4 連接 GitHub Repo

先在 GitHub 建立一個空 repo，例如 my-first-site，不要勾 README，避免第一次 push 時產生版本差異。

git remote add origin https://github.com/你的帳號/my-first-site.git
git push -u origin main

第一次 git push 時，Windows 可能會跳出 GitHub 登入視窗或瀏覽器授權畫面。這是正常的，代表 GitHub 正在確認你有權限把本機 commit 上傳到該 repo。

6.5 到 GitHub 啟用 Pages

進入 repo → Settings → Pages，設定：

Source: Deploy from a branch
Branch: main
Folder: /root

儲存後等待發布完成。

8. 第一次 Push 的 GitHub 認證與 Branch 設定

第一次在 Windows 執行 git push 時，學生最容易卡在兩件事：

  1. GitHub 要求登入或授權，不知道該輸入什麼。
  2. 本機 branch 名稱和 GitHub repo 的預設 branch 不一致，例如本機是 master，GitHub 是 main。
這一段非常重要

git commit 只是在本機建立版本紀錄；git push 是要把版本上傳到 GitHub，所以 GitHub 必須驗證你是誰、你有沒有權限寫入這個 repo。

7.1 為什麼 push 需要認證？

GitHub 需要確認三件事：

  • 你是不是這個 GitHub 帳號本人。
  • 你對這個 repo 有沒有寫入權限。
  • 如果是 Private repo，你是否被授權存取。
本機 commit ↓ git push GitHub 檢查登入身分與 repo 權限 ↓ 通過 commit 上傳到 GitHub repo ↓ GitHub Pages 讀取指定 branch / folder 並重新部署網站

7.2 Windows 新手建議：用 Git Credential Manager 登入

在 Windows 安裝 Git for Windows 時，通常會一起安裝或搭配 Git Credential Manager。第一次 git push 時，它可能會跳出 GitHub 登入視窗或開啟瀏覽器要求授權。

工具 指令 是否通常隨 Git for Windows 安裝？ 用途
Git for Windows git 是 版本控制本體，例如 git add、git commit、git push。
Git Credential Manager 通常由 git 背後呼叫 通常是 協助 HTTPS push/fetch 時跳出瀏覽器登入，並安全保存 GitHub 憑證。
GitHub CLI gh 否，需要另外安裝 GitHub 官方命令列工具，可用 gh auth login、gh auth status 管理 GitHub 登入狀態。

常見流程如下：

  1. 執行 git push -u origin main。
  2. 跳出 GitHub 登入視窗，或開啟瀏覽器。
  3. 登入 GitHub 帳號。
  4. 如果帳號有雙重驗證，完成 2FA。
  5. 授權 Git Credential Manager 存取 GitHub。
  6. 回到命令列，push 繼續完成。

如果成功，之後同一台電腦通常不會每次都要求登入，因為認證會被 Windows 憑證管理機制保存。

檢查目前 Git 是否有 credential helper

git config --global credential.helper

如果看到類似 manager 或 manager-core，代表 Git 會使用憑證管理工具。

7.3 如果命令列問 username / password，要輸入什麼？

如果你使用 HTTPS 遠端網址，例如：

https://github.com/你的帳號/my-first-site.git

push 時可能會問：

Username for 'https://github.com':

這裡輸入你的 GitHub username。

接著如果問：

Password for 'https://你的帳號@github.com':

不要輸入 GitHub 帳號密碼。GitHub 已不建議用帳號密碼做命令列 Git 操作。這裡應該改用 Personal Access Token，或改用 Git Credential Manager / GitHub CLI 完成登入。

Token 要當成密碼保管

Personal Access Token 可以視為命令列用的替代密碼。不要把 token 寫進 index.html、README.md、程式碼、截圖或教學投影片中。

7.4 Personal Access Token 是什麼？

Personal Access Token，簡稱 PAT，可以理解成「給命令列或工具使用的授權密碼」。它可以設定有效期限與權限範圍，比直接使用帳號密碼更安全。

新手只需要先知道：

  • push 到 repo 需要寫入權限。
  • token 要保密，外洩就要立刻刪除或重建。
  • token 建議設定到期日。
  • 權限不要開太大，只給需要的 repo 與必要權限。
情境 建議處理方式
Windows 第一次 push，跳出瀏覽器登入 照流程登入 GitHub，讓 Git Credential Manager 保存認證。
命令列要求 password 不要輸入 GitHub 密碼，改輸入 Personal Access Token。
學校電腦或共用電腦 不建議保存長期認證；用完後登出 GitHub，必要時清除 Windows Credential Manager 內的 GitHub 憑證。
未來要更正式開發 可以學 SSH key 或另外安裝 GitHub CLI。新手第一階段先用 HTTPS + Git Credential Manager 即可。

7.5 可選方式：使用 GitHub CLI 登入

GitHub CLI 需要另外安裝。它的指令是 gh，不是 Git for Windows 內建的 git 指令。安裝後可以用以下指令登入：

gh auth login

如果你有另外安裝 GitHub CLI，新手可以依照畫面選擇：

  • GitHub.com
  • HTTPS
  • Login with a web browser

登入完成後，再執行：

git push -u origin main

7.6 第一次是否要設定 branch？

答案是：建議第一次就明確設定成 main，尤其是用命令列 git init 建 repo 的時候。

原因是 GitHub 現在常見的預設 branch 名稱是 main，但有些 Git 環境、本機設定或舊教學仍可能出現 master。如果本機和 GitHub 的 branch 名稱不同，新手第一次 push 很容易出現困惑。

建議寫法一：初始化時直接指定 main

git init -b main

這個方式會在初始化 repo 時，直接把初始 branch 設為 main。

建議寫法二：初始化後改成 main

git init
git branch -M main

git branch -M main 的意思是：把目前本機 branch 重新命名成 main。這不是進入進階多人協作，也不是要求學生現在就學 branch flow，只是把主線名稱統一。

git branch -M main 是必須的嗎？

不是每一種情況都絕對必須，但如果你目前在 master，而 GitHub repo 或 GitHub Pages 預設使用 main，就很建議執行。它的作用是把本機目前這條 branch 直接改名成 main，讓本機和遠端名稱一致。

git branch -M main 拆開看是什麼意思？

部分 意思
git branch 管理本機 branch 的指令。
-M 強制重新命名目前 branch。即使 main 已存在，也會以強制方式處理。
main 新的 branch 名稱。現在 GitHub 新 repo 常見預設 branch 名稱是 main。

什麼情況需要執行？

目前情況 要不要執行 git branch -M main？ 原因
git status 顯示 On branch master 建議執行 把本機 master 改成 main，避免 push 到錯的 branch，或 GitHub Pages 讀不到。
git status 已經顯示 On branch main 不用執行 本機已經是 main，不需要再改名。
你是從 GitHub clone 下來的 repo 通常不用 clone 時通常會自動跟著遠端 repo 的預設 branch。
你是本機從零開始 git init 建議執行，或直接用 git init -b main 避免本機初始 branch 叫 master，但 GitHub 預設是 main。

判斷方式很簡單，先看目前 branch：

git status
git branch

如果看到：

On branch master

或：

* master

就可以執行：

git branch -M main

再確認一次：

git status
git branch

應該會變成：

On branch main

或：

* main

為什麼改成 main 後才跳出 GitHub 認證？

這通常代表前一次 git push 還沒真正進入「對 GitHub 寫入遠端 branch」的認證流程，就已經因為本機 branch 名稱或推送目標不一致而失敗。

例如你本機在 master：

On branch master

但你執行的是：

git push -u origin main

這時 Git 可能會找不到本機的 main，或無法正確建立本機 branch 與遠端 branch 的追蹤關係。當你執行：

git branch -M main

本機目前 branch 就被改名成 main。接著再執行：

git push -u origin main

Git 這時就能明確知道：「我要把本機 main 推到遠端 origin/main。」因此流程會正式連到 GitHub，接著才觸發瀏覽器登入或 Git Credential Manager 認證。

main 是 GitHub Pages 的建議標準，不是 Git 本體的硬性規定

Git 本身不強制你的主要 branch 一定要叫 main。它可以叫 master、dev、website，甚至其他名稱。但 GitHub 新 repo 與 GitHub Pages 教學通常以 main 作為預設主分支。為了讓本機、GitHub repo、GitHub Pages 發布來源一致，新手教學建議統一使用 main。

什麼時候 main 幾乎是必要的？

如果 GitHub Pages 設定是：

Source: Deploy from a branch
Branch: main
Folder: /root

那你的網站檔案就應該 push 到遠端的 main branch。否則你可能把檔案推到 master，但 GitHub Pages 卻在讀 main，網站就不會更新。

所以更精確地說：

  • main 不是 Git 的硬性規定。
  • 但如果 GitHub repo 預設 branch 是 main，GitHub Pages 也設定讀 main，那本機就建議也統一成 main。
  • git branch -M main 的作用，就是把本機目前 branch 名稱整理成 main。

檢查目前在哪個 branch

git branch

你會看到類似：

* main

星號 * 代表你目前所在的 branch。

7.7 如果我之前都沒設定 branch，為什麼也能用？

因為很多情況下 GitHub 或你的工具已經幫你處理好了。例如：

  • 你直接在 GitHub 網頁建立 repo，GitHub 會建立預設 branch。
  • 你使用 GitHub Desktop 或 VS Code，有些流程會自動對應預設 branch。
  • 你 clone 既有 repo 時，Git 會直接跟著遠端 repo 的預設 branch。

所以不是每次都一定要手動設定 branch；只是如果你是從本機 git init 開始建立專案，為了避免 master / main 不一致，建議一開始就明確設定。

7.8 如何檢查目前是否已經認證？

如果學生問：「我怎麼知道這台電腦已經登入 GitHub，可以 push 了？」可以依照下面順序檢查。

第一步：確認 remote 是 HTTPS 還是 SSH

在專案資料夾執行：

git remote -v

如果看到類似：

origin  https://github.com/你的帳號/你的repo.git (fetch)
origin  https://github.com/你的帳號/你的repo.git (push)

代表這個 repo 使用的是 HTTPS 連線。Windows 新手大多會走這種方式，通常會透過 Git Credential Manager、GitHub CLI 或 Personal Access Token 完成認證。

如果看到類似：

origin  git@github.com:你的帳號/你的repo.git (fetch)
origin  git@github.com:你的帳號/你的repo.git (push)

代表這個 repo 使用的是 SSH key 認證。

第二步：檢查 Git 是否啟用 Credential Manager

如果你使用 HTTPS，可以執行：

git config --global credential.helper

如果看到：

manager

或：

manager-core

代表 Git 會透過 Windows 的 Git Credential Manager 管理 GitHub 登入。

注意

這只代表 Git 有啟用憑證管理工具，不代表你一定已經登入成功。真正是否能存取 repo，還要用 git fetch 或 git push 測試。

第三步：用 git fetch 測試是否能讀取 repo

git fetch

如果沒有錯誤，通常代表你至少能讀取這個 repo。若是 Private repo，這通常也代表你的認證有成功。

如果出現下面這類訊息，就比較像認證或權限問題：

Authentication failed
Invalid username or password
Permission denied
Repository not found
403

但要注意：git fetch 成功只代表讀取權限正常，不一定代表你一定有 push 權限。最終能不能寫入，仍要看 git push。

第四步：如果有另外安裝 GitHub CLI，才檢查 gh 登入狀態

gh auth status 不是 Git 內建指令，必須先另外安裝 GitHub CLI 才能使用。如果沒有安裝，這一步可以跳過。

gh auth status

如果已登入，會看到類似：

Logged in to github.com account 你的帳號
Git operations protocol: https
Token: *******************

如果已安裝 GitHub CLI 但尚未登入，可以執行：

gh auth login

然後依照畫面選擇 GitHub.com、HTTPS、Login with a web browser。

第五步：Windows 圖形介面檢查

可以到 Windows 搜尋：

認證管理員

或英文系統搜尋：

Credential Manager

進入 Windows 認證，找是否有類似：

git:https://github.com
github.com
GitHub

如果有，代表 Windows 目前保存了 GitHub 相關憑證。但憑證也可能是舊帳號、過期 token 或錯誤授權；如果一直認證失敗，可以刪掉相關 GitHub 憑證後重新 git push 觸發登入。

如果使用 SSH，檢查 SSH 認證

如果 git remote -v 顯示的是 git@github.com:...，可以執行：

ssh -T git@github.com

成功時會看到類似：

Hi 你的帳號! You've successfully authenticated, but GitHub does not provide shell access.

這代表 SSH key 認證成功。

推薦的新手檢查順序

git remote -v
git config --global credential.helper
git fetch

如果有另外安裝 GitHub CLI，再加：

gh auth status

如何判斷是不是認證問題？

錯誤訊息關鍵字 比較可能的原因
Authentication failed GitHub 認證失敗，可能沒登入、token 錯誤或憑證過期。
Invalid username or password 帳號或 token 錯誤。不要輸入 GitHub 帳號密碼，應改用瀏覽器授權、GitHub CLI 或 Personal Access Token。
Permission denied / 403 已連到 GitHub，但沒有該 repo 的權限，或認證帳號不是 repo 擁有者。
Repository not found repo URL 寫錯、repo 不存在，或 Private repo 沒權限。
failed to push some refs 這句本身只是總結，可能是遠端已有 commit、本機 branch 不一致，也可能是其他 push 被拒絕原因；要看它上面幾行真正錯誤。
remote contains work that you do not have locally 遠端 GitHub repo 已經有 README、.gitignore 或其他 commit，本機沒有先同步；這不是認證問題。

7.9 本篇推薦的新手完整指令

如果是本機從零開始建立網站 repo，建議直接用這組：

mkdir my-first-site
cd my-first-site

git init -b main

echo "<h1>Hello GitHub Pages</h1>" > index.html

git add .
git commit -m "Add initial static website"

git remote add origin https://github.com/你的帳號/my-first-site.git
git push -u origin main

如果 git init -b main 在你的 Git 版本不能用，就改用：

git init
git branch -M main

9. GitHub Pages 設定說明：為什麼這樣設？

設定項目 建議值 原因
Source Deploy from a branch 對入門靜態網站最簡單。只要 push 到指定 branch，GitHub 就會自動發布。
Branch main main 是主要開發線。新手先不需要額外建立 gh-pages branch。
Folder /root 網站檔案直接放 repo 最外層，最容易理解，也最不容易找不到首頁。
首頁檔名 index.html 多數網站伺服器與靜態網站平台會把 index.html 當作預設首頁。

7.1 什麼時候會用 /docs？

當 repo 主要是程式專案，但你只想把文件網站公開時，可以把網站放在 /docs。

my-project/
├── src/
│   └── main.py
├── tests/
│   └── test_main.py
└── docs/
    ├── index.html
    └── style.css

此時 GitHub Pages 設定可以選：

Branch: main
Folder: /docs

意思是：程式碼仍在 repo 裡，但網站只從 docs 資料夾發布。

7.2 什麼時候會用 gh-pages branch？

gh-pages 是很多前端專案或早期教學常見的發布分支。概念是把「原始碼」和「要發布的網站成品」分開。

分支 用途
main 放原始碼，例如 React / Vue 專案原始碼
gh-pages 放 build 後的靜態網站成品

但對剛學 HTML / CSS / JS 的學生來說，先不要把流程弄太複雜。第一階段建議使用 main + /root。

10. 網站更新流程

網站不是只發布一次。之後你修改 HTML、CSS 或 JS 後，要重新 commit 並 push。

8.1 日常更新流程

# 查看修改狀態
git status

# 加入修改
git add .

# 建立 commit
git commit -m "Update homepage layout"

# 上傳到 GitHub
git push

push 後，GitHub Pages 會重新部署。等部署完成，重新整理網站就會看到新版本。

8.2 常見 commit message

修改內容 建議 commit message
新增首頁 Add homepage
調整版面 Update homepage layout
新增 CSS Add site styles
修正圖片路徑 Fix image path
補上專案說明 Update README with website URL
GitHub Pages 的觀念

不是你在本機改完就會上線，而是要 commit + push 到 GitHub，GitHub Pages 才有機會拿到新檔案並重新發布。

11. Public / Private 與網站可見性

GitHub Pages 發布出來的網站，本質上是拿來被瀏覽器公開讀取的。因此學生應該養成一個觀念：只要是發布成網站的內容，就要假設別人可能看得到。

Repo 類型 適合情境 注意事項
Public Repo 作品集、靜態網頁練習、課堂展示 repo 與網站內容都公開，適合新手練習 GitHub Pages。
Private Repo 未公開草稿、內部文件、私人練習 即使 repo 是 private，也不要把敏感資料放進要發布的網站內容。
不要放進網站的東西
  • 密碼
  • API Key
  • 資料庫連線字串
  • 學生個資或真實成績資料
  • 未授權公開的圖片、影片、文件

12. 常見問題與排除

10.1 網站打開是 404

常見原因：

  • GitHub Pages 還沒部署完成。
  • 沒有 index.html。
  • index.html 沒放在設定的資料夾中。
  • Settings → Pages 的 branch 或 folder 選錯。
  • repo 名稱或網址打錯。

10.2 CSS 沒有套用

通常是路徑錯誤。假設結構是：

my-first-site/
├── index.html
└── style.css

HTML 應該寫：

<link rel="stylesheet" href="style.css">

如果結構是：

my-first-site/
├── index.html
└── css/
    └── style.css

HTML 應該寫：

<link rel="stylesheet" href="css/style.css">

10.3 圖片不見

注意大小寫。Windows 對檔名大小寫通常比較寬鬆，但網站部署到伺服器後，Profile.jpg 和 profile.jpg 可能會被視為不同檔案。

<!-- 檔名是 profile.jpg，就不要寫成 Profile.JPG -->
<img src="images/profile.jpg" alt="profile">

10.4 更新後網站還是舊的

可能原因：

  • 你忘記 push。
  • GitHub Pages 還在部署。
  • 瀏覽器快取還沒更新，可以按 Ctrl + F5 強制重新整理。
  • 你改的是另一個 branch，不是 Pages 指定的 branch。

10.5 我可以用 GitHub Pages 做登入系統嗎？

不建議。GitHub Pages 不會替你跑後端伺服器，也不適合直接處理帳密登入、資料庫寫入或金流。它比較適合前端展示、文件、作品集與靜態頁面。

13. 練習題：做一個自己的第一個靜態網站

練習 A：個人介紹頁

  1. 建立 repo：my-profile-site。
  2. 建立 index.html。
  3. 內容包含姓名、科系、目前學習的程式語言。
  4. 啟用 GitHub Pages：main + /root。
  5. 把網站網址貼到 README。

練習 B：三頁式小網站

建立以下結構：

my-school-project/
├── index.html
├── about.html
├── projects.html
└── style.css

首頁加入連結：

<nav>
  <a href="index.html">首頁</a>
  <a href="about.html">關於我</a>
  <a href="projects.html">作品</a>
</nav>

至少做三次 commit：

git commit -m "Add homepage"
git commit -m "Add about and projects pages"
git commit -m "Add basic site styles"

11.1 完成標準

  • GitHub repo 看得到所有檔案。
  • Settings → Pages 顯示網站網址。
  • 網站打開不是 404。
  • 首頁、CSS、連結都正常。
  • commit message 看得出每次修改目的。

14. 本篇重點整理

  • Git 是版本控制工具；GitHub 與 Gitea 是 Git 遠端代管平台。
  • GitHub Pages 是靜態網站發布功能。
  • 它需要你指定發布來源：branch + folder。
  • 新手最推薦：main branch + /root folder。
  • index.html 是首頁，建議放 repo 最外層。
  • 修改網站後要 git add、git commit、git push。
  • 第一次 push 可能會要求 GitHub 認證；Windows 新手建議先用 Git for Windows 內建/搭配的 Git Credential Manager。GitHub CLI 是可選的額外工具。
  • 可以用 git remote -v、git fetch 檢查 Git 連線；若有另裝 GitHub CLI，才可用 gh auth status。
  • 本機從零開始建立 repo 時，建議第一次就明確使用 main branch。
  • git branch -M main 的作用是把本機目前 branch 改名成 main，通常用來處理本機是 master、遠端是 main 的不一致。
  • main 不是 Git 的硬性規定，但若 GitHub Pages 設定讀取 main，本機就應統一推到 main。
  • 網站上線後，公開內容不要放密碼、token、個資或未授權素材。
