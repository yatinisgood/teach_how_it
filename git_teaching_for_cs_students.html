1. 為什麼資工新生要先學 Git？

在學校剛開始寫程式時，很多人會用「複製資料夾」來備份，例如：

作業_v1
作業_v2
作業_最後版
作業_最後版_真的最後
作業_最後版_修正版

這種方式在小作業還可以勉強撐住，但一旦進入真正的專案環境，會很快失控。Git 的作用就是把每一次重要修改記錄起來，讓你知道：

  • 誰改了什麼？
  • 什麼時間改的？
  • 為什麼要改？
  • 如果改壞了，能不能回到以前的版本？
現實工作環境的重點

公司、開源專案、研究室、多人專題幾乎都會使用 Git 或類似的版本控制流程。越早熟悉，越能避免「程式碼不見、版本混亂、不知道自己改了什麼」這些新手常見問題。

2. 先懂 5 個核心概念

名詞 白話解釋 你可以想成
Repository / Repo 一個專案的版本倉庫，通常對應一個程式專案。 一個有歷史紀錄的專案資料夾
Commit 把目前修改正式記錄成一個版本。 遊戲存檔，但會附上說明
Check in 口語上常指「把改好的東西提交進版本控管」。在 Git 裡通常就是 add + commit，有時也包含 push。 把作業交進系統
Push 把本機的 commit 上傳到 GitHub。 把本機存檔同步到雲端
Pull 把 GitHub 上的新版本抓回本機。 從雲端更新本機資料夾
先記這句話

git add 是選擇要收進版本的檔案，git commit 是建立版本紀錄，git push 是上傳到 GitHub。

3. 建立 GitHub 帳號

Git 是版本控制工具，GitHub 是常見的雲端程式碼平台。你可以先把 GitHub 想成「程式碼版的 Google Drive + 歷史紀錄 + 專案協作平台」。

前往 GitHub

打開 https://github.com/，選擇 Sign up 建立帳號。

選擇帳號名稱

建議使用乾淨、可長期使用的名稱，例如英文名、暱稱、學號縮寫。避免太隨便的名字，因為未來可能會出現在履歷、作品集或專題連結中。

設定 Email

建議使用你會長期保留的 Email。Git commit 也會綁定作者名稱與 Email。

開啟雙重驗證

GitHub 帳號未來可能會放作業、專題、作品集，建議啟用 Two-factor authentication，避免帳號被盜。

提醒

不要把密碼、API Key、資料庫帳密、學校系統帳密放進 GitHub。即使是 Private repo，也不該把敏感資訊直接 commit 進去。

4. Windows 環境安裝 Git

4.1 安裝 Git for Windows

  1. 前往 Git 官方網站：https://git-scm.com/downloads/win
  2. 下載 Windows 版本安裝檔。
  3. 安裝時大多數選項可以維持預設值。
  4. 安裝完成後，開啟 Git Bash 或 Windows Terminal。

4.2 確認 Git 是否安裝成功

在 Git Bash、PowerShell 或 Windows Terminal 輸入：

git --version

如果看到類似以下結果，代表成功：

git version 2.xx.x.windows.x

4.3 建議安裝 VS Code

資工新生建議同時安裝 Visual Studio Code，因為之後寫 C、Python、JavaScript、網頁、資料處理都常用得到。

  • VS Code 可以直接看到 Git 修改狀態。
  • 可以用圖形化方式 stage、commit、push。
  • 但初學仍建議先學會基本命令列，理解背後發生什麼事。

5. 第一次設定 Git 身分

Git 需要知道每一次 commit 是誰做的。第一次使用 Git 時，請設定你的名稱與 Email。

git config --global user.name "你的名字"
git config --global user.email "你的Email@example.com"

例如：

git config --global user.name "Sherlock Tsai"
git config --global user.email "student@example.com"

確認設定：

git config --global --list
這不是 GitHub 登入

user.name 和 user.email 是寫進 commit 紀錄的作者資訊，不是輸入 GitHub 密碼。GitHub 登入驗證會另外處理，例如瀏覽器授權、GitHub CLI、Personal Access Token 或 SSH。

6. 第一次建立 Repo：手動用 GitHub 網頁操作

這是最適合新手的第一種方式。先用網頁建立 repo，再把檔案上傳上去。

6.1 建立 Public Repo

登入 GitHub

進入 GitHub 首頁，按右上角的 +，選擇 New repository。

輸入 Repository name

例如：my-first-python-project。建議用英文、小寫、單字間用 -。

選擇 Public

Public 代表任何人都可以看見這個 repo。適合放作品集、練習專案、開源範例，但不要放隱私資料。

勾選 Add a README file

README 是專案說明書。新手建議勾選，讓 repo 一開始就有第一個檔案。

按 Create repository

建立完成後，你會看到一個新的 GitHub 專案頁面。

6.2 建立 Private Repo

流程幾乎一樣，只是在 Visibility 選擇 Private。

Private repo 只有你自己與被授權的人看得到。適合：

  • 學校作業
  • 還沒公開的作品
  • 有個人資料或尚未整理乾淨的練習專案
Private 不是保險箱

Private repo 仍然不應該放密碼、金鑰、資料庫連線字串。真正的敏感資訊應該放在環境變數、祕密管理工具或本機設定檔中，並用 .gitignore 排除。

6.3 用 GitHub 網頁上傳檔案

  1. 進入 repo 頁面。
  2. 按 Add file。
  3. 選擇 Upload files。
  4. 拖曳檔案或選擇檔案。
  5. 在下方 Commit changes 區塊填寫 commit message。
  6. 按下 Commit changes。

範例 commit message：

Add first Python homework file

這種方式很直覺，但缺點是比較不像實際工程環境。現實開發通常會在本機寫程式，再用 Git 指令或 VS Code 上傳。

7. 第一次建立 Repo：使用命令列

7.1 情境 A：GitHub 已經建立空 repo，本機再推上去

這是最常見的新手流程。你先在 GitHub 建立 repo，再把本機專案推上去。

假設你在 GitHub 建立了一個 repo：

https://github.com/你的帳號/my-first-project

在本機建立專案資料夾

mkdir my-first-project
cd my-first-project

建立第一個檔案

echo "# My First Project" > README.md

初始化 Git

git init

查看目前狀態

git status

你會看到 README.md 是尚未追蹤的檔案。

加入版本控管

git add README.md

如果你想一次加入目前資料夾內所有修改：

git add .

建立第一個 commit

git commit -m "Initial commit"

先檢查本機目前 branch 名稱

在第一次 push 前，先不要急著推送。請先確認本機目前在哪個 branch：

git status
git branch

如果看到：

On branch master

或：

* master

代表本機目前主要 branch 叫 master。

如果看到：

On branch main

或：

* main

代表本機目前主要 branch 已經叫 main。

設定主要分支名稱

git branch -M main

這行的作用是把本機目前的 branch 重新命名成 main。如果你剛剛 git status 看到 On branch master，但接下來要執行 git push -u origin main，就應該先做這一步，避免本機分支名稱與推送目標不一致。

如果已經是 main，就不用再改

如果 git status 已經顯示 On branch main，可以略過 git branch -M main，直接繼續設定 remote 與 push。

連接 GitHub 遠端 repo

git remote add origin https://github.com/你的帳號/my-first-project.git

推送到 GitHub

git push -u origin main
-u 是什麼？

git push -u origin main 會設定本機 main 分支預設對應 GitHub 的 main。之後通常只要輸入 git push 就可以。

7.2 情境 B：本機已經有一個作業資料夾

假設你已經有一個資料夾：

C:\Users\你的名字\Desktop\python-homework

打開 Git Bash 或 Windows Terminal，進入該資料夾：

cd /c/Users/你的名字/Desktop/python-homework

或在檔案總管進入資料夾後，按右鍵選擇 Open Git Bash here。

接著執行：

git init
git add .
git commit -m "Add first homework files"
git branch -M main
git remote add origin https://github.com/你的帳號/python-homework.git
git push -u origin main

7.3 Public / Private 在命令列有差嗎？

對 Git 指令來說，Public 和 Private 的操作幾乎一樣。差別在 GitHub 權限：

  • Public repo：別人可以讀取與 clone。
  • Private repo：只有被授權的人可以讀取與 clone。

你如果沒有權限 push，GitHub 會拒絕上傳。

常見但容易誤判的狀況

如果 git status 顯示 On branch master，但你執行 git push -u origin main，可能會在 branch 名稱不一致時先失敗，還沒真正進入 GitHub 認證。先執行 git branch -M main，再重新 git push -u origin main。

8. 第一次 Push 前的 main 分支與認證謎題

第一次用命令列把本機 repo 推到 GitHub 時，很多新手會遇到一個很容易誤判的狀況：

error: failed to push some refs to 'https://github.com/你的帳號/你的repo.git'

看到這句話時，不一定代表你還沒登入 GitHub。它也可能代表：本機 branch 名稱和你要推送的 branch 名稱不一致，導致 Git 還沒真正走到 GitHub 認證階段。

8.1 為什麼 git branch -M main 後才跳出認證？

假設你本機目前狀態是：

git status

看到：

On branch master

但你執行的是：

git push -u origin main

這時 Git 的意思會變成：「請把本機的 main 推到遠端 origin/main。」可是你的本機目前其實叫 master，不叫 main，因此可能還沒正式進入 GitHub 認證流程，就先因為推送目標不一致而失敗。

當你執行：

git branch -M main

本機目前 branch 會從 master 改名成 main。接著再執行：

git push -u origin main

Git 就能明確知道：「我要把本機 main 推到遠端 origin/main。」這時流程才會正式連到 GitHub，因此可能會跳出瀏覽器登入或 Git Credential Manager 認證視窗。

原本： 本機 branch = master 你執行 git push -u origin main → 本機沒有明確的 main 可推 → 可能先在 branch / ref 階段失敗 執行 git branch -M main 後： 本機 branch = main 你執行 git push -u origin main → 本機 main 對遠端 origin/main → 正式連到 GitHub → 觸發登入 / 認證

8.2 git branch -M main 是必須的嗎？

精確說法是：不是 Git 本身強制必須，但新手第一次推 GitHub repo 時很建議做。

Git 本身不規定主要 branch 一定要叫 main。它可以叫 master、dev、website 或其他名稱。但 GitHub 新 repo 常見預設 branch 是 main，很多教學與平台設定也會用 main。因此如果本機還叫 master，但遠端與教學都使用 main，就容易出現第一次 push 卡住的狀況。

8.2.0 為什麼 GitHub 現在常用 main？是必須的嗎？

main 不是 Git 的硬性規定。Git 本身允許 branch 叫任何合法名稱，例如 master、main、dev、release 都可以。

但 GitHub 近年新建立的 repository 常見預設 branch 名稱是 main，很多教學、範例、GitHub Pages、GitHub Actions 流程也會假設主要 branch 是 main。因此對新手來說，跟著使用 main 可以減少很多不必要的分支名稱混亂。

問題 答案
Git 規定一定要叫 main 嗎？ 沒有。Git 不強制主要 branch 名稱。
GitHub 一定只能用 main 嗎？ 不是。GitHub repo 也可以使用 master 或其他 branch，但預設與新教學常見是 main。
新手為什麼建議用 main？ 讓本機 branch、GitHub repo 預設 branch、教學指令都一致，降低第一次 push 的錯誤率。
什麼時候可以不用改成 main？ 如果你的團隊、公司、Gitea、舊專案明確使用 master，而遠端預設 branch 也是 master，就可以繼續用 master。
教學結論

不是因為 main 比 master 技術上更強，而是因為 GitHub 現代預設與多數新手教學都偏向 main。教學生時，先統一成 main 最不容易出錯。

8.2.1 那如果本機是 master，直接 git push -u origin master 不就好了？

理論上可以。這行的意思是：

git push -u origin master

把本機目前的 master branch 推到遠端，並建立或更新遠端的 origin/master。如果只是要「讓 push 成功」或「把本機 master 上傳到遠端」，它確實可能會解決當下的 push 問題。

但新手教學不優先建議這樣做，原因是它可能造成 GitHub repo 上同時出現兩條主線：

GitHub repo ├── main ← GitHub 預設 branch，README 或網頁設定可能在這裡 └── master ← 你從本機 push 上去的新 branch

這會帶來幾個新手常見困惑：

  • GitHub 頁面預設顯示 main，但你的檔案被推到 master，所以看起來像沒有上傳成功。
  • 如果之後使用 GitHub Pages、Actions 或其他教學設定，它們可能預設讀取 main。
  • repo 裡可能同時存在 main 和 master，新手會不知道哪一條才是正式版本。
  • 之後要合併、改預設 branch 或刪除多餘 branch，反而增加教學複雜度。
做法 結果 適合情境
git push -u origin master 把本機 master 推到遠端 master 你明確想繼續使用 master 當主分支，且遠端平台也設定使用 master。
git branch -M main
git push -u origin main		 先把本機主分支改名成 main，再推到遠端 main GitHub 新手教學、GitHub repo 預設 branch 是 main、或之後可能使用 GitHub Pages。
教學建議

如果是初學者第一次建立 GitHub repo，建議統一成 main。不是因為 Git 不能用 master，而是為了讓「本機 branch、GitHub 預設 branch、教學指令」三者一致，降低第一次 push 的錯誤率。

情況 是否需要 git branch -M main？ 原因
git status 顯示 On branch master 建議執行 把本機主分支改名成 main，和 GitHub 新 repo 的常見預設一致。
git status 已經顯示 On branch main 不用執行 本機已經是 main，直接 push 即可。
你是從 GitHub clone 下來的 repo 通常不用 clone 時通常會跟著遠端 repo 的預設 branch。
你是本機從零開始 git init 建議執行，或直接用 git init -b main 避免本機初始 branch 叫 master，但遠端 GitHub 用 main。

8.3 git branch -M main 拆開看

部分 意思
git branch 管理本機 branch 的指令。
-M 強制重新命名目前 branch。也就是把目前這條分支改成新的名字。
main 新的 branch 名稱。

8.4 第一次 push 建議標準流程

如果你是本機從零開始建立 repo，建議流程改成：

git init

# 先檢查目前 branch 名稱
git status
git branch

# 如果目前是 master，建議統一改成 main
git branch -M main

git add .
git commit -m "Initial commit"
git remote add origin https://github.com/你的帳號/你的repo.git
git push -u origin main

如果你的 Git 版本支援，也可以一開始就指定初始 branch：

git init -b main
git add .
git commit -m "Initial commit"
git remote add origin https://github.com/你的帳號/你的repo.git
git push -u origin main
這次解開的謎題

如果 git branch -M main 之後，再執行 git push -u origin main 才跳出 GitHub 認證，通常代表前面不是單純「沒有認證」，而是本機 branch 還在 master，和你要推送的 main 不一致。先統一 branch 名稱後，Git 才能真正進入遠端 push 與認證流程。

8.5 如果還是 failed to push some refs？

如果已經確認本機是 main，但還是看到：

failed to push some refs

要看錯誤訊息上方幾行。如果有：

remote contains work that you do not have locally

通常代表 GitHub repo 上已經有 README、.gitignore 或其他 commit，本機沒有先同步。可以用：

git pull origin main --allow-unrelated-histories
git push -u origin main

如果錯誤訊息是：

Authentication failed
Invalid username or password
Permission denied
Repository not found
403

那才比較像是認證、權限或 remote URL 的問題。

可以直接 push master 嗎？

可以使用 git push -u origin master 把本機 master 推上遠端，但這會建立或使用遠端 master。若 GitHub 預設 branch 是 main，新手可能會以為檔案沒有出現。因此本教學建議先用 git branch -M main 統一本機主分支，再推 main。

9. 上傳檔案後的 Check in / Commit 作業

9.1 標準日常流程

日常寫程式時，最基本流程如下：

# 1. 查看目前有哪些檔案被修改
git status

# 2. 選擇要提交的檔案
git add 檔名

# 或一次加入所有修改
git add .

# 3. 建立 commit，寫清楚這次做了什麼
git commit -m "你的備註"

# 4. 上傳到 GitHub
git push

9.2 一個完整例子

假設你新增了 hello.py：

print("Hello, Git!")

接著執行：

git status
git add hello.py
git commit -m "Add hello world Python example"
git push

9.3 Check in 到底是什麼？

在很多公司或前輩口語中，會說「你 check in 了嗎？」意思通常是：

  • 你有沒有把修改加入版本控管？
  • 你有沒有 commit？
  • 你有沒有 push 到遠端 repo？

嚴格來說，Git 的正式指令是 commit 和 push，但在口語中 check in 常常泛指「把修改交進版本控制系統」。

不要每改一個字就 commit

Commit 應該是一個有意義的小單位，例如「完成登入畫面」、「修正計算錯誤」、「新增 README」。太碎或太大都不好。

10. Commit Message 怎麼寫？

Commit message 是給未來的自己、同學、助教、同事看的。好的 message 可以讓人一眼知道這次修改的目的。

10.1 不好的寫法

update
fix
test
123
修改
最後版
不知道

10.2 比較好的寫法

Add student score calculator
Fix division by zero error
Update README with setup steps
Refactor input validation logic
Add sample data for homework 2

10.3 建議格式

新手可以先用這種格式：

動詞 + 修改對象 + 目的
情境 建議 commit message
新增第一份作業 Add homework 1 solution
修正輸出格式 Fix output format for homework 1
補上專案說明 Update README with usage instructions
移除不需要的測試檔 Remove unused test file
簡單原則

看到 commit message 的人，應該不用打開程式碼，就大概知道你這次做了什麼。

11. Public / Private 怎麼選？

類型 誰看得到？ 適合放什麼？ 不適合放什麼？
Public 任何人 作品集、開源練習、可展示專案、教學範例 作業答案、個資、API Key、未整理的私人程式
Private 你與被授權的人 學校作業、個人練習、還沒公開的專題 密碼、金鑰、機密資料、正式個資資料集

11.1 資工新生建議

  • 作業：用 Private。
  • 自己整理過、確定可以展示的小作品：用 Public。
  • 想放履歷的作品：Public，但 README 要寫清楚。
  • 含有資料集、帳密、token、個資：不要直接上傳。

11.2 README 要寫什麼？

12. 常見錯誤與排除

12.1 git 不是內部或外部命令

代表 Windows 找不到 Git。請確認：

  • Git 是否已安裝。
  • 安裝時是否有加入 PATH。
  • 關掉 Terminal 後重新開啟。

12.2 push 時要求登入或失敗

可能原因：

  • 你尚未登入 GitHub。
  • 你沒有該 repo 的寫入權限。
  • Private repo 需要正確驗證。
  • 遠端 URL 寫錯。

查看遠端 URL：

git remote -v

修改遠端 URL：

git remote set-url origin https://github.com/你的帳號/你的repo.git

12.3 nothing to commit, working tree clean

這通常不是錯誤，意思是目前沒有新的修改可以 commit。可能情況：

  • 你真的沒有改檔案。
  • 你改的檔案沒有存檔。
  • 你在錯的資料夾執行 Git 指令。

12.4 accidentally committed wrong file

如果只是還沒 push，新手可以先不要急著處理複雜指令，先詢問老師、助教或有經驗的人。尤其是如果誤 commit 密碼或 token，重點不是刪掉檔案而已，還要立即撤換該密碼或 token。

12.5 忘記先 pull 就 push

如果 GitHub 上有別人更新過，你本機沒有同步，push 可能失敗。先執行：

git pull

再依照提示處理。初學階段若是個人 repo，通常比較少遇到複雜衝突。

13. 練習題：完成你的第一個 GitHub 專案

練習 A：Private 作業 Repo

  1. 建立一個 Private repo，名稱為 cs-homework-practice。
  2. 在本機建立 README.md。
  3. 新增一個 hello.py。
  4. commit message 寫：Add hello world homework practice。
  5. push 到 GitHub。

練習 B：Public 作品 Repo

  1. 建立一個 Public repo，名稱為 my-first-portfolio-project。
  2. README 說明這個專案是你的第一個 Git 練習。
  3. 新增一個簡單程式，例如 BMI 計算器、成績平均計算器或猜數字遊戲。
  4. 至少做兩次 commit。
  5. 到 GitHub 頁面確認檔案與 commit 紀錄都有出現。

14. 給家長或老師的教學建議

第一次教 Git，不建議一開始就講 branch、merge、rebase、fork、pull request。那些是團隊協作與開源流程，很重要，但對剛接觸程式的新生來說容易過載。

第一階段只要讓學生建立以下心智模型即可：

  • 一個專案一個 repo。
  • 寫完一小段有意義的修改，就 commit。
  • commit message 要讓未來的自己看得懂。
  • push 之後，GitHub 上應該看得到最新版本。
  • Public 可以展示，Private 適合作業。

等學生能熟練完成 add → commit → push，再進入 branch、pull request、code review，會比較接近真實工程教育的節奏。
