GitHub CLI / gh 入門教學

用指令列操作 GitHub：從 Git、GitHub、Gitea 到 GitHub CLI

這份教學適合已經大概知道 git add、git commit、git push，但還不熟 GitHub CLI 的使用者。重點會放在觀念釐清、安裝、登入、建立 repo、clone、issue、PR，以及常見排錯。

1. 先搞懂 Git / GitHub / Gitea / GitHub CLI 2. 安裝 GitHub CLI 3. 登入與驗證 4. 常用工作流程 5. 常用指令速查 6. 常見問題與排錯

1. 先搞懂 Git / GitHub / Gitea / GitHub CLI

Git

Git 是版本控制系統。它是免費、開源、分散式的版本控制工具，用來追蹤檔案變更、管理分支、合併程式碼、回到歷史版本。

你在本機執行的 git init、git status、git commit、git push 都是 Git 指令。

GitHub

GitHub 是以 Git 為核心的雲端協作平台。它幫你託管 repo，並提供 Pull Request、Issue、Actions、Release、權限控管等協作功能。

簡單講：Git 管版本；GitHub 管協作、託管與工作流程。

Gitea

Gitea 是可自架的 Git 服務。你可以把它想成「自己架一套類 GitHub 的服務」，常見於 NAS、內網、公司私有環境。

它不是取代 Git 指令，而是提供 Git repository 的 server 端管理介面。

那 GitHub CLI 是什麼？

GitHub CLI 的指令是 gh，它讓你在 Terminal / PowerShell / CMD 裡操作 GitHub 功能，例如建立 repo、開 PR、看 issue、查 workflow、產生 release，而不需要一直切到瀏覽器。

觀念重點：git 與 gh 不是同一個東西。git 處理版本控制本身；gh 處理 GitHub 平台功能。實務上兩者會一起用。
工具主要功能典型指令
Git本機版本控制、分支、commit、merge、push/pullgit status
git commit
git push
GitHub雲端 repo、PR、Issue、Actions、Release、團隊協作通常在網頁操作
Gitea自架 Git server 與協作平台搭配 git remote、git push
GitHub CLI用指令列操作 GitHub 平台功能gh auth login
gh repo create
gh pr create

2. 安裝 GitHub CLI

GitHub CLI 支援 Windows、macOS、Linux。安裝前建議先確認 Git 已安裝：

git --version

如果沒有 Git，請先安裝 Git，再安裝 GitHub CLI。因為 GitHub CLI 很多流程會搭配本機 Git repository 使用。

Windows：使用 WinGet

在 PowerShell 或 Windows Terminal 執行：

winget install --id GitHub.cli --source winget

升級：

winget upgrade --id GitHub.cli --source winget
注意：Windows 安裝器會修改 PATH。安裝後請重新開啟一個新的 Windows Terminal 視窗；只開新分頁有時不會吃到新的 PATH。

macOS：使用 Homebrew

brew install gh

升級：

brew upgrade gh

Ubuntu / Debian：使用官方 apt 來源

(type -p wget >/dev/null || (sudo apt update && sudo apt install wget -y)) \
  && sudo mkdir -p -m 755 /etc/apt/keyrings \
  && out=$(mktemp) && wget -nv -O$out https://cli.github.com/packages/githubcli-archive-keyring.gpg \
  && cat $out | sudo tee /etc/apt/keyrings/githubcli-archive-keyring.gpg > /dev/null \
  && sudo chmod go+r /etc/apt/keyrings/githubcli-archive-keyring.gpg \
  && sudo mkdir -p -m 755 /etc/apt/sources.list.d \
  && echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null \
  && sudo apt update \
  && sudo apt install gh -y

升級：

sudo apt update
sudo apt install gh

確認是否安裝成功

gh --version

若能看到版本號，就代表 gh 指令已可使用。

3. 登入與驗證

安裝完 GitHub CLI 後，第一件事是登入 GitHub：

gh auth login

常見選項可以這樣選：

提示建議選擇說明
What account do you want to log into?GitHub.com一般 GitHub 帳號選這個；企業私有 GitHub 才選其他 host。
Preferred protocol for Git operations?HTTPS 或 SSH新手通常 HTTPS 較直覺；已配置 SSH key 者可選 SSH。
Authenticate Git with your GitHub credentials?Yes可讓 git push / git pull 使用 GitHub CLI 管理的認證。
How would you like to authenticate?Login with browser透過瀏覽器登入最簡單。

登入後檢查狀態：

gh auth status

4. 常用工作流程

A. 建立一個新的 GitHub repo

情境：你本機已經有一個資料夾，想要變成 Git repo 並推到 GitHub。

cd my-project
git init
git add .
git commit -m "Initial commit"
gh repo create my-project --private --source=. --remote=origin --push

說明：

  • --private：建立私有 repo。若要公開，可改 --public。
  • --source=.：使用目前資料夾。
  • --remote=origin：自動把 GitHub repo 加成 origin remote。
  • --push：建立後直接 push。

B. Clone 一個 GitHub repo

gh repo clone OWNER/REPO

例如：

gh repo clone cli/cli

C. 查看目前 repo 資訊

gh repo view

打開網頁版：

gh repo view --web

D. Issue 操作

gh issue list
gh issue view 123
gh issue create --title "登入頁錯誤" --body "重現步驟：..."

E. Pull Request 操作

典型 PR 流程：

建立分支
修改檔案
commit
push
建立 PR
git checkout -b feature/login-fix
# 修改檔案
git add .
git commit -m "Fix login error"
git push -u origin feature/login-fix
gh pr create --title "Fix login error" --body "修正登入頁錯誤"

查看 PR 狀態：

gh pr status

列出 PR：

gh pr list

切到某個 PR 的分支進行測試：

gh pr checkout 123

F. GitHub Actions 操作

gh run list
gh run view
gh run watch

這對 CI/CD 很有用，可以不用打開瀏覽器就看到 workflow 執行狀態。

G. Release 操作

gh release create v1.0.0 --title "v1.0.0" --notes "第一個正式版本"

6. 常見問題與排錯

問題 1：gh 不是內部或外部命令

通常是 PATH 尚未更新。Windows 請重新開啟新的 Windows Terminal 視窗，再執行：

gh --version

問題 2：git push 還是要求帳密

先檢查 GitHub CLI 登入狀態：

gh auth status

如果你登入時沒有讓 GitHub CLI 管理 Git 認證，可以重新登入：

gh auth logout
gh auth login

登入過程中看到「Authenticate Git with your GitHub credentials?」時選 Yes。

問題 3：repo 建好了，但 push 到錯的 branch

先確認本機分支名稱：

git branch --show-current

再確認 remote：

git remote -v

如果本機分支是 main，通常推：

git push -u origin main

如果本機分支是 master，通常推：

git push -u origin master
GitHub 現在新 repo 常見預設分支是 main。但 Git 本身不強制一定要叫 main；你的 repo 可以使用 master、dev 或其他名稱。重點是本機分支、遠端分支與預設分支要對得起來。

問題 4：我要用 Gitea，可以用 GitHub CLI 嗎？

通常不適合。GitHub CLI 是針對 GitHub.com / GitHub Enterprise 的工具；Gitea 雖然也是 Git server 與協作平台，但它不是 GitHub。對 Gitea repo 進行版本控制時，主要仍使用 git clone、git remote、git push、git pull。

git clone https://你的-gitea-server/user/repo.git
cd repo
git status
git push

問題 5：我要先學 Git 還是 GitHub CLI？

建議順序：

  1. 先學 Git 基本：status、add、commit、branch、merge、push、pull。
  2. 再學 GitHub 基本：repo、issue、pull request、actions。
  3. 最後學 GitHub CLI：把 GitHub 網頁操作搬到命令列，提高效率。

結論

Git 是版本控制核心；GitHub 是雲端協作平台；Gitea 是可自架的 Git 服務；GitHub CLI 則是把 GitHub 的平台功能帶進命令列。

日常開發時可以這樣分工：版本變更用 git，GitHub 平台互動用 gh。熟悉後，你可以少開很多網頁，尤其在建立 repo、開 PR、看 issue、查 Actions 狀態時非常方便。
